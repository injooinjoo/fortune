# P5-B8 Review: Dark-only app lock-in

Verdict: **PASS**

## Checks

### 1. Apple Guideline 4.5 (dark-only concern)
No blocker. HIG does not mandate that apps support light mode — many brand apps (Netflix, TikTok, Spotify) are dark-only. What reviewers care about is *manifest consistency* (declared support must match runtime behavior). Flipping `userInterfaceStyle: 'automatic'` → `'dark'` *reduces* the risk of a 4.5/2.5 flag because the app now declares what it actually renders.

### 2. Info.plist regeneration
Confirmed. `@expo/prebuild-config` plugin at
`node_modules/@expo/prebuild-config/build/plugins/unversioned/expo-system-ui/withIosUserInterfaceStyle.js:38-46`
maps `'dark'` → `UIUserInterfaceStyle = Dark` via `createInfoPlistPluginWithPropertyGuard`. Next `expo prebuild` (EAS cloud build) will rewrite the existing `<key>UIUserInterfaceStyle</key><string>Automatic</string>` at `ios/app/Info.plist:93-94` to `Dark`. The property guard still permits a manual override in Info.plist, but since the current plist value was itself auto-generated from the prior `'automatic'` setting, there is no conflict.

### 3. StatusBar behavior with `UIViewControllerBasedStatusBarAppearance = false`
Safe. `ios/app/Info.plist:95-96` has `UIViewControllerBasedStatusBarAppearance = false` — this is the Expo default set in `@expo/config-plugins/build/plugins/withIosBaseMods.js:139`. In this mode the status bar is controlled application-wide via the legacy `UIApplication` API, which is exactly what expo-status-bar's `<StatusBar style="light" />` uses under the hood (`RNCStatusBarManager.setStyle`). The Info.plist `UIStatusBarStyle = UIStatusBarStyleDefault` is only the *initial* value before JS mounts; the runtime call from `_layout.tsx:62` overrides it to light content after hydration. No change needed here.

### 4. Splash screen flash
Low risk. Splash `backgroundColor: '#0B0B10'` matches `fortuneTheme.colors.background`. The native launch storyboard → JS handoff keeps the same dark canvas, so there is no white flash window. Splash icon is already authored for dark background (in-use for months).

### 5. System controls side effects
Positive. With `UIUserInterfaceStyle = Dark`, iOS will render system keyboards, action sheets, share sheets, date pickers in dark variants matching the app. This is an improvement over `Automatic`, which previously followed the device setting and could produce light system chrome inside a dark app on light-mode devices.

### 6. Hardcoded light backgrounds
Found 9 `#FFFFFF` backgroundColor usages across 5 files. None are critical light screens — all are intentional white pill/tile fills on dark surfaces:
- `src/components/social-auth-pill-button.tsx:34,39,43` — social login pills (brand-correct)
- `src/screens/signup-screen.tsx:215,262` — email/phone login pills (matches social pills)
- `src/features/chat-surface/chat-surface.tsx:963,993` — inline chip fills
- `src/features/fortune-results/heroes/hero-birthstone.tsx:93` — birthstone highlight
- `src/features/ios-widgets/primitives/star-field.tsx:88` — star dot color (widget)

Parent `<Screen>` containers all use `fortuneTheme.colors.background` (dark). No regression introduced by this change.

### 7. navigationTheme
`theme.ts:57` extends React Navigation `DarkTheme`. Consistent with manifest `'dark'`.

### 8. Signup / paywall (reviewer-visible)
- `signup-screen.tsx` — renders inside `<Screen>` (dark bg); white pills on dark canvas. Dark-compliant.
- `premium-screen.tsx:227` — renders inside `<Screen>`; all text uses `AppText` with `fortuneTheme.colors.*`. Dark-compliant.

## Notes
- No code changes required beyond the two already in place.
- After next EAS build, verify Info.plist in the built IPA shows `UIUserInterfaceStyle = Dark`. Existing local `ios/app/Info.plist` will be regenerated by prebuild; committing the stale `Automatic` value is harmless because prebuild overwrites it.
- Existing `#FFFFFF` debt is pre-existing and intentional; not a regression.
