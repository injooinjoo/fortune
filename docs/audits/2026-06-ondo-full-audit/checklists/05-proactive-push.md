# 공통 운영 원칙

- 온도 앱 전체 검토의 일부로 이 체크리스트를 사용한다.
- 1차 작업은 **QA/리뷰/증거 수집/수정 제안**이 목적이다.
- 코드 수정은 하지 않는다. 수정은 취합 후 별도 Fix Agent 작업으로 진행한다.
- 모든 발견 항목은 P0/P1/P2/P3로 분류한다.
- 단순 의견이 아니라 증거 기반으로 보고한다.
- 문서와 코드가 다르면 코드/실제 동작을 우선 증거로 삼는다.
- 시뮬레이터 성공을 실기기 성공으로 간주하지 않는다.
- 서버 작업 완료를 유저 화면 성공으로 간주하지 않는다.

## 심각도 기준

- P0: 앱 사용 불가, 결제/토큰 손실, 보안/개인정보 문제, App Store 즉시 리젝 가능
- P1: 핵심 기능 실패, 사용자 신뢰 크게 저하, 주요 전환/매출 손상
- P2: UX 불편, 특정 경로에서 실패, 반복 버그 가능성
- P3: 개선 제안, polish, 미세 최적화

# 5. Proactive Push Reviewer

## 추가 원칙
- Push notification은 실기기 검증 필요 여부를 반드시 분리한다.

## 검토 목표
- 캐릭터가 먼저 보내는 메시지와 푸시 알림이 안정적으로 생성/저장/표시/이동되는지 확인한다.
- 푸시는 왔는데 방에 없거나, DB에는 있는데 알림이 없거나, 과도하게 발송되는 문제를 찾는다.

## 체크리스트

### 1. 선톡 발송 조건
- 아침/점심/저녁/굿나잇/부재 체크 조건이 명확한가?
- 과도한 발송 방지가 있는가?
- 유저 알림 설정/선호가 반영되는가?
- 같은 메시지가 반복 발송되지 않는가?

### 2. 메시지 내용
- 시간대와 상황에 맞는가?
- 캐릭터 관계 톤이 자연스러운가?
- 반복 문구나 generic 문구가 많은가?
- 부담스럽거나 스팸처럼 느껴지지 않는가?

### 3. 이미지 선톡
- 이미지 reveal 규칙이 안전한가?
- Storage URL이 유효한가?
- 이미지 만료/실패 fallback이 있는가?
- reveal claim 보안 문제가 없는가?

### 4. Push notification
- foreground에서 필요한 알림이 표시되는가?
- 같은 채팅방이 열려 있을 때 suppress 정책이 맞는가?
- proactive type 예외 처리가 있는가?
- background push 수신 후 클릭 시 올바른 방으로 이동하는가?

### 5. DB/상태 / 실기기
- proactive message row가 저장되는가?
- notification receipt 추적이 가능한가?
- claimed/revealed/delivered 상태가 일관적인가?
- 리스트 preview와 방 내용이 일치하는가?
- push token 등록, Expo push receipt, foreground/background 동작, 앱 종료 상태 클릭 이동 검증 필요 여부
