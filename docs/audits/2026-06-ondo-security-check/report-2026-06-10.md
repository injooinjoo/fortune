## P1

### P1-1. `fortune-tarot` is still reachable without auth and can proceed to LLM generation with valid payload
- **Impact**: App policy currently says all AI/fortune paths consume tokens. `fortune-tarot` explicitly falls back to `anonymous`, and production no-auth POST reaches function code. With valid `selectedCards`, it reaches `LLMFactory.createFromConfig('tarot')` without server auth/token charge. This is a cost/revenue leak if public endpoint is abused.
- **Evidence**:
  - `supabase/config.toml:41-42` sets `[functions.fortune-tarot] verify_jwt = false`.
  - `supabase/functions/fortune-tarot/index.ts:420-423`: JWT is optional; no-auth becomes `anonymous`.
  - `supabase/functions/fortune-tarot/index.ts:494-511`: after selected card validation, LLM is called.
  - Production smoke: no-auth empty POST returned app-level `400 {"success":false,"error":"필수 필드 누락: selectedCards"}`, proving gateway did not block before function execution.
- **Minimal fix direction**: either require auth + token reserve before LLM, or if guest tarot is intentionally free, add server-side anonymous quota/device/IP abuse guard and make BM docs/UI explicit.
- **Post-fix verification**: no-auth valid tarot payload returns 401/402/quota before LLM; authenticated insufficient-token user returns 402; authenticated enough-token user creates consume transaction before LLM.

### P1-2. Private Storage is fixed at bucket level, but generated images are still persisted as expiring signed URLs only
- **Impact**: Privacy exposure is improved, but result/chat/push payloads can break after 7 days because stored `imageUrl` is a signed URL, not stable `{bucket,path}` metadata with refresh. This is data durability and support risk for past-life/poster/talisman images.
- **Evidence**:
  - Production bucket query shows private buckets: `palm-reading-images=false`, `poster-guide-images=false`, `past-life-portraits=false`, `talisman-images=false`, `yearly-encounter-images=false`, `friend-avatars=false`, `character-audio-messages=false`.
  - `supabase/functions/generate-poster-guide/index.ts:64` sets 7-day TTL.
  - `supabase/functions/generate-poster-guide/index.ts:246-256` returns only `signed.signedUrl`.
  - `supabase/functions/process-poster-jobs/index.ts:136-156` persists `imageUrl`/`result_image_url` from that signed URL.
  - Similar 7-day signed URL patterns remain in `fortune-past-life` and `generate-talisman`.
- **Minimal fix direction**: return/store canonical `{bucket, path, expiresAt}` plus transient signed URL; mobile refreshes signed URL on render/push hydration.
- **Post-fix verification**: after manually expiring/replacing signed URL, old result reopens and image refreshes from bucket/path; direct public bucket URL stays 401/403.

### P1-3. Real secrets exist in local `.env.local` / `.env`; file mode was hardened during this QA
- **Impact**: Not a git leak (`.gitignore` covers `.env*`), but local workstation risk is high because service role/JWT/API keys are present. Initial mode was `0644`, so any local user/process with read access could exfiltrate.
- **Evidence**:
  - `.env.local` exists, size 3006, initially mode `0644`, contains sensitive-like key names including Supabase service role/JWT secret, OpenAI, Stripe, Toss, Upstash, Sentry token names. Values were not printed in this report.
  - `.env` exists, size 869, initially mode `0644`, contains Supabase/API key names.
  - `git check-ignore -v .env .env.local` confirms ignored; `git ls-files` lists only `.gitignore`, not env files.
  - **Immediate remediation done**: `chmod 600 .env .env.local`; verification now shows `.env 600`, `.env.local 600`.
- **Residual risk**: rotate any secrets previously exposed to tools/artifacts/logs; keep secret scanner pre-commit/CI.
- **Post-fix verification**: no tracked secret; secret scanner clean.

### P1-4. `generate-poster-guide` top comment is stale and contradicts current server-charge design
- **Impact**: Not runtime security by itself, but future agents may reintroduce client-side charge-before-generation because the file header says token charge is client responsibility.
- **Evidence**:
  - `supabase/functions/generate-poster-guide/index.ts:9` says token charge is client `completeSurvey` responsibility.
  - Actual safe path: `start-poster-job/index.ts:101-117` uses `schedule_poster_job_with_charge`; `process-poster-jobs/index.ts:91-93` rejects uncharged jobs; `generate-poster-guide/index.ts:350-354` requires worker auth.
- **Minimal fix direction**: update header comment to “worker-only; charge/reserve happens in `start-poster-job` RPC before queueing.”
