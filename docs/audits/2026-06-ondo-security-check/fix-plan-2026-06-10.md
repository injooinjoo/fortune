## Phase 1 — P1 Fix: Lock Down `fortune-tarot` Before LLM

### Task 1.1: Decide policy: no guest tarot for now

**Decision:** Because `AGENTS.md` states “모든 AI 기능/운세는 토큰 소비,” the safe default is **auth required + token consume before LLM**. Do not add guest quota unless product explicitly wants free guest tarot later.

**Implementation rule:** no-auth valid tarot payload must return `401` before cohort pool, percentile, or LLM work.

### Task 1.2: Add auth guard to `fortune-tarot`

**Objective:** Prevent unauthenticated callers from reaching any cost-bearing or data-producing path.

**Files:**
- Modify: `supabase/functions/fortune-tarot/index.ts`
- Optional modify: `supabase/config.toml`

**Steps:**
1. Replace the optional fallback:
   ```ts
   const userId = (await deriveUserIdFromJwt(req)) ?? 'anonymous'
   ```
   with a required-user guard:
   ```ts
   const userId = await deriveUserIdFromJwt(req)
   if (!userId) {
     return new Response(
       JSON.stringify({ success: false, error: '로그인이 필요해요.', errorCode: 'auth_required' }),
       { status: 401, headers: { ...corsHeaders, 'Content-Type': 'application/json' } },
     )
   }
   ```
2. Keep `verify_jwt=false` only if CORS/preflight or custom parsing requires function-level handling; otherwise set `[functions.fortune-tarot] verify_jwt = true` and verify mobile caller behavior. Conservative first pass: function-level guard is enough and avoids accidental platform auth/CORS regression.
3. Remove/adjust comments that mention anonymous cohort logging.

**Verification:** no-auth request fails before `selectedCards` validation with `401 auth_required`, not `400 selectedCards`.

### Task 1.3: Consume token atomically before LLM/cache-miss result

**Objective:** Align tarot with the business rule that AI/fortune consumes finite tokens.

**Files:**
- Modify: `supabase/functions/fortune-tarot/index.ts`
- Read: `supabase/migrations/20260505150000_token_atomic_rpcs_and_text_reference.sql`
- Read: `supabase/functions/_shared/fortune-pricing-generated.ts` or current pricing source

**Steps:**
1. Determine tarot token cost from generated pricing / app constants. Do not hardcode a guessed value.
2. Before the LLM call, call `consume_token_atomic` via the service/admin Supabase client:
   ```ts
   const chargeReferenceId = `fortune-tarot:${crypto.randomUUID()}`
   const { data: consumeResult, error: consumeError } = await supabaseClient.rpc('consume_token_atomic', {
     p_user_id: userId,
     p_cost: tarotCost,
     p_description: '타로 운세 사용',
     p_reference_type: 'fortune_tarot',
     p_reference_id: chargeReferenceId,
     p_idempotency_key: chargeReferenceId,
   })
   ```
3. If `consumeError` indicates insufficient balance, return `402` with `errorCode: 'insufficient_tokens'` before LLM.
4. Include `chargeReferenceId` / consume transaction id in server logs without exposing secrets.
5. If LLM fails after token consume, use existing `refund_token_atomic` pattern (same shape as `process-poster-jobs`) to refund exactly once.
6. Decide cohort-hit behavior explicitly:
   - Preferred: cached/cohort results still consume app tokens because user receives an AI fortune product.
   - If product wants free cache hits later, document it as a separate policy exception; do not implement silently.

**Verification:**
- Authenticated insufficient-token user gets `402` and no LLM call.
- Authenticated enough-token user gets a consume transaction before LLM.
- Forced LLM failure refunds exactly once.
- Replayed same request/idempotency does not double-consume if an idempotency key is reused.

## Phase 2 — P1 Fix: Canonical Private Storage Metadata + Signed URL Refresh

### Task 2.1: Define shared media metadata contract

**Objective:** Make bucket/path the source of truth and signed URL a transient cache.

**Files:**
- Create or modify: `supabase/functions/_shared/private_media.ts`
- Create or modify: `apps/mobile-rn/src/lib/private-media.ts`

**Contract:**
```ts
export type PrivateMediaRef = {
  bucket: string;
  path: string;
  signedUrl?: string;
  signedUrlExpiresAt?: string;
};
```

**Rules:**
- Server persists `bucket` + `path` + `signedUrlExpiresAt` alongside legacy `imageUrl` for backward compatibility.
- Mobile treats `imageUrl` as cache/fallback only.
- Push payloads should prefer identifiers and avoid bearer signed URLs when possible.

## Phase 4 — Controlled Write-Path QA Evidence

### Task 4.1: AdMob SSV QA

**Objective:** Prove rewarded ad grant integrity in production/sandbox.

**Prerequisites:** AdMob test SSV callback or signed callback fixture; test Supabase user.

**Acceptance:**
1. Valid SSV callback grants exactly once.
2. Replay same `transaction_id` grants zero additional tokens.
3. Client POST remains `403 ssv_required`.
4. Ledger row exists with expected `transaction_id`.

**Evidence:** sanitized curl command, response status, DB row IDs, before/after balance.

### Task 4.2: StoreKit sandbox QA

**Objective:** Prove purchase finalization does not lose or double-grant tokens.

**Acceptance:**
1. Consumable purchase grants exact token amount once.
2. Cancel does not grant.
3. Replay/restore owned transaction returns safe replay state without double grant.
4. Subscription activation/restore grants monthly finite tokens according to current policy.

**Evidence:** real-device/TestFlight or sandbox logs, transaction ID redacted, DB transaction rows, balance before/after.

### Task 4.3: Delete-account QA

**Objective:** Prove account deletion removes user data and generated Storage prefixes.

**Acceptance:**
1. Test account can request deletion.
2. DB rows listed in `delete-account/index.ts` are removed or anonymized as designed.
3. Storage objects under all user prefixes are removed.
4. Auth user is deleted and re-login fails.
5. No unrelated user data is touched.

**Evidence:** test user id, row counts before/after, Storage object counts before/after, auth lookup result.
