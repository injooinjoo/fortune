## Executive verdict

### iOS App Store Review
**NO-GO for final App Review submission** until existing P0/P1 manual evidence is closed.

Primary blockers:
- Real iPhone clean-install rejection path evidence pending.
- IAP success/cancellation/restore evidence pending.
- iPad review path evidence pending.
- App Store Connect build processing/final submission evidence pending.

### BM / monetization
**High risk / fail against BM v2.2 as written.**

Primary blockers:
- BM says all AI/fortune should consume tokens, but code still contains subscriber-unlimited and daily/chat free bypasses.
- Ad reward can be double-granted through SSV GET + client POST and lacks transaction-level idempotency.
- Purchase token grant is not atomic.
- Subscription restore path likely conflicts with new verified purchase gate.

### UX/UI / button/runtime
**No P0 found in static audit, but several P1/P2 product/runtime issues require fixes before a confident store-quality release.**

Primary risks:
- Hero result-card tap may open detail screen without payload, showing fallback/placeholder instead of actual result.
- Paid fortune result can be generated before token charge succeeds in some paths.
- Async poster queue can be created before billing completes.
- Premium purchase unavailable state disables the CTA without the intended explanatory handler.

## P0 findings

### P0-1. iOS release evidence package is still incomplete

Minimum resolution:

### P0-2. Subscribers bypass token consumption despite BM v2.2 token model
Evidence:
- `.claude/docs/22-business-model.md:7`, `:27-35`
- `AGENTS.md:237-244`
- `supabase/functions/_shared/token_charge.ts:37-50`, `:67-76`

Minimum resolution:
- Decide between true monthly token allocation model vs unlimited subscription model. If BM v2.2 remains source of truth, remove `hasUnlimitedSubscription` charge skip and update UI copy.

### P0-3. Ad reward grant can duplicate and race
Evidence:
- `apps/mobile-rn/src/lib/ad-rewards.ts:3-10`, `:226-228`
- `supabase/functions/grant-ad-reward/index.ts:3-7`, `:45-66`, `:78-103`, `:124-164`, `:193-215`
- `supabase/migrations/20260503193000_ad_reward_log.sql:4-16`

Impact:
- SSV GET and client POST can both grant. Daily cap is not transactionally enforced.

Minimum resolution:
- Prefer SSV-only when active. Add unique transaction id/signature constraints and atomic grant RPC.

### P0-4. Purchase token grant is not atomic
Evidence:
- `supabase/functions/payment-verify-purchase/index.ts:726-749`, `:807-857`, `:873-894`
- `supabase/migrations/20260505120000_token_transactions_purchase_unique.sql:1-13`

Impact:
- Race/retry can update balance before unique transaction insert fails, creating over-credit without ledger integrity.

Minimum resolution:
- Move verified purchase → balance update → transaction insert → first-purchase bonus into one atomic DB RPC.

### P1-6. Some fortune result paths generate/queue before billing is settled

### P1-7. Premium unavailable purchase CTA is disabled without explanatory handler
